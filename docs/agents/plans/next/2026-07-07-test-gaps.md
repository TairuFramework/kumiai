## Medium

- ~~Nothing exercises `@kumiai/broadcast` or `@kumiai/rpc` against the real `hub-server`;
  `tests/integration/` holds only `hub-relay.test.ts` and `hub-tunnel-echo.test.ts`.~~
  **Largely done** on `feat/app-lane-delivery`. `tests/integration/test/` now holds nine
  files, including `app-lane-delivery.test.ts`, `directed-lane.test.ts`,
  `hub-log-lane.test.ts`, `mls-permissions.test.ts`, and `wire-hub-smoke.test.ts` — the
  rpc peer runs over a real `createHub`. The residue: `hub-tunnel-echo.test.ts` still uses
  an in-memory `HubLike` double, so tunnel-specific real-hub semantics remain untested.
  Note also that this suite is type-checked by the repo gate but NOT run by it (see
  `tests/integration/package.json`) — it must be run by hand for any exported-signature
  change.
- No concurrency tests at the peer: `resync()` racing an inbound Commit, `dispose()`
  during in-flight handshake, `to()` before init. No test of interleaved
  `encrypt`/`processMessage` on one `GroupHandle`, nor `decrypt` receiving a commit (the
  mutate-then-throw path). (Companions:
  `next/2026-07-07-mls-state-serialization-secret-hygiene.md`,
  `backlog/2026-07-07-rpc-peer-lifecycle-hardening.md`.)
- No mixed error/success anycast test (`responder.test.ts:30` tests only a lone erroring
  responder) — would catch the error-suppression bug. (Companion:
  `next/2026-07-07-anycast-soundness.md`.)
- `hub-server/test/hub.test.ts` never exercises the `hub/receive` `after` cursor,
  multi-page backlog drain (>50 messages), abort-cleanup, or the publish-during-drain
  window. Key-package limits (`assertKeyPackageFetchAllowed`, `maxCount` capping,
  fetch-for-nonexistent-DID) entirely untested. (Companions:
  `next/2026-07-07-hub-receive-lifecycle.md`,
  `next/2026-07-07-hub-keypackage-subscribe-caps.md`.)
- `CommitRejectedError`'s captured payload (`proposals`, `senderLeafIndex`, the
  `wrapCommitPolicy` capture path) never asserted — tests only check `instanceof`.
- **Two guards in the app-lane drain survive deletion with the suite green** (`@kumiai/rpc`).
  Carried over from the `feat/app-lane-delivery` review; the first was re-checked by inversion on
  2026-07-21 and is still uncovered — deleting `peer.ts:1575` leaves all 338 rpc tests passing.
  - `peer.ts:1575`, `retentionOf(protocols[name], prc) !== 'log'` — the only thing standing between
    a hostile member and *"make every returning member re-fire an ephemeral handler out of the
    log"*, the hazard retention classes exist to prevent. Untested because every `retain: 'log'`
    publish in the suite names an already-logged procedure; a test has to publish a logged frame
    naming an **ephemeral** one, which no honest dispatch would do. Note the drain test at
    `peer-app-drain.test.ts:176` does not reach it — that ephemeral frame rides the mailbox lane
    and the fake hub filters it out of `fetchTopic`, so the line is never evaluated.
  - `peer.ts` self-echo guard — structurally unreachable in every test: a peer must publish while
    its segment is still unloaded for its own frame to land in its own buffer, and every test
    flushes an empty seed pull first.
- `hub-tunnel/test/` — no test asserts the teardown contract: `session-end` frame
  published, `hub.unsubscribe` called, `onSessionEnd` firing on a peer's frame.
  (Companion: `backlog/2026-07-07-hub-tunnel-reliability.md`.)
