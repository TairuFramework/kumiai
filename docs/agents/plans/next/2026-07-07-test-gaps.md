# Close test gaps

**Priority:** 7 — persist→restore path, real-hub integration, concurrency tests.
**Origin:** 2026-07-02 audit (commit `bb343d9`), milestone
`milestones/2026-07-audit-remediation.md`.

Several of these are natural companions to the fix docs in this milestone (noted inline);
write them with the fixes where possible, sweep the rest here.

## High

- **The persist→restore path has zero tests anywhere:** `restoreGroup`
  (`mls/src/group.ts:454`), `encodeClientState`, `decodeClientState` (`codec.ts:12-18`) —
  including whether a restored handle can still encrypt/decrypt across epochs. This is
  what every real client does on restart.

## Medium

- Nothing exercises `@kumiai/broadcast` or `@kumiai/rpc` against the real `hub-server`;
  `tests/integration/` holds only `hub-relay.test.ts` and `hub-tunnel-echo.test.ts`, and
  the tunnel "integration" test uses an in-memory `HubLike` double — real-hub semantics
  (sender exclusion, auth, durability/ack) are never tested against the peer. Fix:
  group-RPC-over-real-hub integration test wiring `createHubTunnelTransport` through the
  rpc hub-mux over a real `createHub`.
- No concurrency tests at the peer: `resync()` racing an inbound Commit, `dispose()`
  during in-flight handshake, `to()` before init. No test of interleaved
  `encrypt`/`processMessage` on one `GroupHandle`, nor `decrypt` receiving a commit (the
  mutate-then-throw path). (Companions:
  `next/2026-07-07-mls-state-serialization-secret-hygiene.md`,
  `backlog/2026-07-07-rpc-peer-lifecycle-hardening.md`.)
- No mixed error/success anycast test (`responder.test.ts:30` tests only a lone erroring
  responder) — would catch the error-suppression bug. (Companion:
  `next/2026-07-07-anycast-soundness.md`.)
- `hub-server/test/hub.test.ts` never exercises the `hub/receive` `after` cursor,
  multi-page backlog drain (>50 messages), abort-cleanup, or the publish-during-drain
  window. Key-package limits (`assertKeyPackageFetchAllowed`, `maxCount` capping,
  fetch-for-nonexistent-DID) entirely untested. (Companions:
  `next/2026-07-07-hub-receive-lifecycle.md`,
  `next/2026-07-07-hub-keypackage-subscribe-caps.md`.)
- `CommitRejectedError`'s captured payload (`proposals`, `senderLeafIndex`, the
  `wrapCommitPolicy` capture path) never asserted — tests only check `instanceof`.
- `hub-tunnel/test/` — no test asserts the teardown contract: `session-end` frame
  published, `hub.unsubscribe` called, `onSessionEnd` firing on a peer's frame.
  (Companion: `backlog/2026-07-07-hub-tunnel-reliability.md`.)

## Low

- `sender.test.ts`/`transport.test.ts` use only synchronous `unwrap`; async-unwrap
  ordering untested.
- No acceptor-tunnel leak/idle-teardown test (client vanishing without `session-end`).
- `validateGroupCapability`'s `res: '*'` branch, expired-token path, and `decrypt` fed
  wire-form application-message bytes untested.
- Purge scheduling in `createHub` (`hub.ts:85-94`) untested (interval fires, stops after
  dispose).
- e2e-expo app exercises only `@kumiai/mls`; no mobile-runtime coverage of hub/rpc.
